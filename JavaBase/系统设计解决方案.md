

# 一、单点登录（SSO）核心基础

## 1. 核心定义

单点登录（SSO，Single Sign On）：用户一次登录，多子系统互通，无需重复输入账号密码，核心是“将认证中心独立，所有业务系统仅信任该认证中心”。

## 2. 关键角色

- 用户（Client）：浏览器/客户端，发起登录请求
- 业务系统（Client System）：A系统、B系统等，依赖认证中心校验登录状态
- 认证中心（SSO Server）：唯一负责账号密码验证、签发登录凭证（Token/Ticket）、校验凭证的核心服务

## 3. 完整标准流程（通用版）

1. 用户访问任意业务系统（如a.com），未登录 → 跳转到SSO认证中心（sso.com）
2. SSO认证中心显示登录页，用户输入账号密码
3. SSO验证账号密码通过，创建全局会话（Global Session），并生成凭证（Ticket/Token）
4. SSO重定向回原业务系统，并携带凭证（如：http://a.com/callback?ticket=xxx）
5. 业务系统拿着凭证调用SSO校验接口，校验通过后，业务系统创建本地会话（Local Session），用户登录成功
6. 用户访问其他业务系统（如b.com），未登录 → 跳转SSO
7. SSO通过自身域名下的Cookie，发现已有全局会话，无需重新登录，直接生成新凭证并跳转回该业务系统
8. 业务系统校验凭证通过，自动登录成功

## 4. 关键细节：SSO如何发现已有全局会话？

核心：依赖SSO自身域名下的Cookie，与业务系统无关。

- 用户首次登录SSO成功后，SSO会在服务端创建全局会话（如存入Redis），同时给浏览器种下“属于sso.com域名”的Cookie（如SSO_SESSION=abc123）
- 后续用户访问其他业务系统，跳转SSO时，浏览器会自动携带该Cookie
- SSO服务器拿到Cookie，查询自身存储的全局会话，确认有效后，直接签发凭证，无需重新登录

## 5. SSO与权限的分工（核心边界）

- 认证（Authentication）：“你是谁”，校验账号密码是否正确 → **SSO的核心职责**
- 授权（Authorization）：“你能干啥”，查询用户角色、菜单、按钮权限 → **一般由业务系统自身或独立权限服务负责，不建议放SSO**

补充：简单内部系统可将角色信息放入JWT/Token中返回，但复杂系统（多业务线、复杂权限）建议拆分独立权限服务，降低SSO耦合度。

# 二、JWT核心知识点（SSO主流凭证格式）

## 1. JWT定义与结构

JWT（JSON Web Token）：一种轻量级、可携带用户信息的凭证，由3部分组成，用“.”分隔，本质是“结构化字符串+签名”。

- 头部（Header）：说明加密算法（如HS256、RS256），Base64编码，无加密（公开可见）
- 载荷（Payload）：存储用户核心信息（如userId、username）、过期时间（exp）、签发者（iss），Base64编码，无加密（公开可见，不可存敏感信息）
- 签名（Signature）：用密钥对“头部+载荷”进行加密生成，核心作用是防伪造、防篡改

完整格式：Header.Base64编码.Payload.Base64编码.Signature（加密后）

## 2. SSO中JWT的生成流程（账号密码→Token）

1. 用户在SSO登录页输入账号密码，通过HTTPS POST请求发送到SSO后端（密码加密传输，不裸奔）
2. SSO后端校验账号密码：查询用户库（MySQL/LDAP），比对加密后的密码（如BCrypt加密），同时检查账号状态（是否禁用、过期）
3. 校验通过后，组装载荷（Payload），包含用户基本信息（userId、username）、过期时间等
4. 用密钥（对称/非对称）对“头部+载荷”进行签名，生成JWT Token
5. SSO返回Token给业务系统/前端，同时种下全局会话Cookie（用于后续自动登录）

## 3. 关键概念：摘要（理解签名的核心）

摘要：将“头部+载荷”通过哈希算法（如SHA256）压缩成一串固定长度的乱码，本质是“内容的数字指纹”，特点如下：

- 原文哪怕改一个字符，摘要会完全不一样
- 不能通过摘要反推原文
- 相同原文，永远生成相同摘要

JWT签名的本质：用密钥对“摘要”进行加密，验证时需解密摘要并与重新计算的摘要对比。

## 4. JWT验证流程（业务系统校验Token）

1. 业务系统拿到JWT，拆分成“头部、载荷、签名”三段
2. 用对应密钥（对称/公钥）解密签名，得到原始摘要A
3. 业务系统用相同的哈希算法，对“头部+载荷”重新计算，得到摘要B
4. 对比摘要A和摘要B：一致则Token合法、未被篡改；不一致则Token伪造或被篡改，拒绝登录

# 三、加密算法（对称vs非对称）核心区别

## 1. 对称加密（如HS256）

### 核心特点

- 加密、解密用**同一个密钥**（如“my-secret-key”）
- 可以反推：知道原文、密文和算法，可暴力反推出密钥；密钥泄露后，任何人都能加密、解密、伪造Token
- 优点：简单、高效、速度快
- 缺点：安全性低，密钥分发风险大（多业务系统无法安全共享密钥）

### 适用场景

简单小项目、单机验证、非核心系统，不适合SSO、微服务等多系统场景。

## 2. 非对称加密（如RS256，公钥+私钥）

### 核心特点

- 密钥是一对：私钥（PRIVATE KEY）+ 公钥（PUBLIC KEY）
- 私钥：绝密，仅SSO认证中心持有，用于签发Token（签名）
- 公钥：公开，可分发给所有业务系统，仅用于验证Token（验签）
- 不可反推：从公钥无法反推出私钥（数学上几乎不可能，依赖大数质因数分解，超级计算机需几百万年以上）
- 优点：安全性极高，公钥可随便分发，无泄露风险
- 缺点：比对称加密复杂、速度稍慢

### 适用场景

SSO、OAuth2、OIDC、微服务鉴权等企业级场景（生产环境首选）。

## 3. 核心区别总结（面试必记）

| 对比维度   | 对称加密（HS256）      | 非对称加密（RS256）     |
| :--------- | :--------------------- | :---------------------- |
| 密钥数量   | 1个（唯一密钥）        | 2个（公钥+私钥）        |
| 密钥用途   | 加密、解密、签名、验签 | 私钥签名，公钥验签      |
| 是否可反推 | 是，易反推、易伪造     | 否，数学上几乎不可能    |
| 安全性     | 低，密钥泄露即崩盘     | 高，公钥泄露无影响      |
| 适用场景   | 小项目、单机验证       | SSO、微服务、企业级系统 |

# 四、SSO主流实现方案（补充）

- CAS：经典企业级方案，基于Session+Ticket票据，支持跨域、单点登出，适合传统老旧系统
- OAuth2.0：互联网主流，侧重授权，适合第三方登录（微信、QQ登录）
- OIDC：OAuth2.0+登录标准化，新增ID Token（JWT格式），适合微服务、前后端分离（Keycloak为主流开源框架）
- JWT共享Token：轻量无状态，适合内部小系统，缺点是不易吊销、不天然支持单点登出
- 同根域名Cookie共享：最简单，仅适合同根域名子系统（如a.company.com、b.company.com）

# 五、核心结论（必记）

1. SSO核心：统一认证中心，一次登录多系统互通，依赖自身域名Cookie识别全局会话
2. JWT核心：头部（算法）+载荷（用户信息）+签名（防伪造），验证本质是“对比摘要”
3. 加密算法：企业级SSO必用非对称加密（公钥验签、私钥签名，不可反推）；对称加密仅用于简单场景
4. 权限分工：SSO负责认证（认人），业务系统/独立权限服务负责授权（分权）
