## 数据库与缓存不一致如何解决

# 解决 MySQL 与 Redis 数据一致性问题：从原理到落地（面试 + 实战双适配）

MySQL 和 Redis 数据一致性的核心矛盾是：**双写操作（库 + 缓存）无法保证原子性**，且缓存有过期、网络异常等不可控因素。解决思路不是追求 “绝对一致”（分布式系统中几乎不可能），而是实现 “业务可接受的最终一致”，以下从「核心原则→主流方案→落地细节→面试话术」层层拆解，既懂原理又能落地，还能应对面试官追问。

## 一、先明确核心原则（面试先立住认知）

1. **一致性≠绝对一致**：99% 的业务（如商品详情、用户信息、订单列表）只需 “最终一致”（秒级 / 分钟级），强一致会牺牲性能，得不偿失；
2. **优先保证 “数据不脏”**：缓存中不能出现错误数据（比 “数据延迟” 更重要）；
3. **核心思路**：要么 “先更库，再处理缓存”，要么 “先处理缓存，再更库”，核心是规避 “并发覆盖” 和 “操作失败” 导致的不一致。

## 二、主流解决方案（按场景优先级排序，附落地代码）

### 方案 1：Cache Aside（缓存旁路）—— 90% 业务的首选（简单、易落地）

这是最经典的方案，核心逻辑：**读走缓存，写走数据库，写操作删缓存而非更缓存**。

#### 1. 核心流程

|  操作  |                             步骤                             |                          核心目的                          |
| :----: | :----------------------------------------------------------: | :--------------------------------------------------------: |
| 读操作 | 1. 先查 Redis 缓存；2. 缓存命中→直接返回；3. 缓存未命中→查 MySQL；4. 将 MySQL 结果写入 Redis（设置过期时间）；5. 返回结果 |          优先用缓存提升读性能，缓存失效时自动重建          |
| 写操作 | 1. 先更新 MySQL 数据库；2. 再删除 Redis 对应缓存（而非更新）；3. （可选）延迟 500ms 再次删除缓存 | 避免更新缓存导致的并发覆盖，删缓存后下次读自动重建最新数据 |

#### 2. 为什么 “删缓存” 而非 “更缓存”？

- 避免并发覆盖：线程 A 更新库→更新缓存，线程 B 同时更新库→更新缓存，可能 A 的缓存覆盖 B 的，导致数据不一致；
- 降低更新成本：若缓存值是多表关联结果（如商品详情 + 库存），更新缓存的成本远高于 “删缓存 + 下次读重建”。

#### 3. 落地代码（Java 示例）

```
import org.springframework.data.redis.core.RedisTemplate;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
public class GoodsService {
    private final GoodsMapper goodsMapper;
    private final RedisTemplate<String, Object> redisTemplate;
    private final static String CACHE_KEY_PREFIX = "goods:";
    private final static long CACHE_EXPIRE = 3600L; // 缓存过期时间1小时

    public GoodsService(GoodsMapper goodsMapper, RedisTemplate<String, Object> redisTemplate) {
        this.goodsMapper = goodsMapper;
        this.redisTemplate = redisTemplate;
    }

    // 读操作：缓存旁路
    public GoodsDTO getGoodsById(Long id) {
        String cacheKey = CACHE_KEY_PREFIX + id;
        // 1. 先查缓存
        GoodsDTO cacheGoods = (GoodsDTO) redisTemplate.opsForValue().get(cacheKey);
        if (cacheGoods != null) {
            return cacheGoods;
        }
        // 2. 缓存未命中，查数据库
        GoodsDO goodsDO = goodsMapper.selectById(id);
        if (goodsDO == null) {
            // 缓存空值，避免缓存穿透（设置短过期时间，如5分钟）
            redisTemplate.opsForValue().set(cacheKey, null, 300L);
            return null;
        }
        GoodsDTO goodsDTO = convertDO2DTO(goodsDO);
        // 3. 写入缓存
        redisTemplate.opsForValue().set(cacheKey, goodsDTO, CACHE_EXPIRE);
        return goodsDTO;
    }

    // 写操作：先更库，再删缓存 + 延迟双删
    @Transactional // 保证MySQL更新原子性
    public void updateGoods(GoodsDTO goodsDTO) {
        String cacheKey = CACHE_KEY_PREFIX + goodsDTO.getId();
        // 1. 更新数据库
        goodsMapper.updateById(convertDTO2DO(goodsDTO));
        // 2. 立即删除缓存
        redisTemplate.delete(cacheKey);
        // 3. 延迟500ms再次删除（解决并发读写入脏数据）
        new Thread(() -> {
            try {
                Thread.sleep(500);
                redisTemplate.delete(cacheKey);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }).start();
    }

    // DO/DTO转换（省略）
    private GoodsDTO convertDO2DTO(GoodsDO goodsDO) { /* ... */ }
    private GoodsDO convertDTO2DO(GoodsDTO goodsDTO) { /* ... */ }
}
```

#### 4. 避坑细节（面试加分）

- **删除缓存失败怎么办？**：写库成功后删缓存失败（如 Redis 宕机），需加「重试机制」—— 将 “缓存 key + 删除指令” 写入 MQ（如 RocketMQ），消费端重试删除，直到成功；
- **延迟双删的时间怎么定？**：根据业务 QPS 调整，一般 100~500ms，需覆盖 “缓存未命中→查库→写缓存” 的耗时；
- **缓存过期时间是兜底**：无论删缓存是否成功，缓存过期后会自动重建，保证最终一致（核心兜底方案）。

#### 5. 适用场景

读多写少的业务（如商品详情、用户信息、医院科室信息），QPS 提升 10 倍以上，一致性保障到秒级。

### 方案 2：Read/Write Through（读写穿透）—— 缓存作为主存储代理

核心逻辑：业务层只与 Redis 交互，Redis 负责同步 MySQL，相当于 “把 Redis 当门面”。

#### 1. 核心流程

|  操作  |                            步骤                             |
| :----: | :---------------------------------------------------------: |
| 读操作 | 1. 读 Redis；2. 未命中→Redis 主动查 MySQL，写入自身后返回； |
| 写操作 |   1. 更新 Redis；2. Redis 同步（同步 / 异步）更新 MySQL；   |

#### 2. 落地要点

- 需自定义 Redis 的扩展逻辑（如 Redis Module），或用中间件（如 Redisson）实现；
- 优点：业务层无需感知 MySQL，代码简化；
- 缺点：Redis 与 MySQL 耦合度高，缓存层逻辑复杂，运维成本高；
- 适用场景：Redis 作为主存储、MySQL 仅做持久化的场景（如实时统计数据）。

### 方案 3：Write Back（写回）—— 高性能但一致性弱

核心逻辑：“先更缓存，异步刷库”，极致提升写性能，但一致性最差。

#### 1. 核心流程

- 写操作：更新 Redis→标记缓存为 “脏数据”→异步批量刷入 MySQL（如缓存满 / 定时触发）；
- 读操作：优先读 Redis，未命中查 MySQL。

#### 2. 落地要点

- 优点：写性能极致（无需同步 MySQL）；
- 缺点：数据丢失风险高（Redis 宕机则脏数据丢失）；
- 适用场景：非核心、允许少量数据丢失的业务（如日志缓存、临时统计数据）。

### 方案 4：分布式事务（最终一致）—— 核心金融业务兜底

在金融账户余额更新的强一致场景中，我用‘分布式事务 + MQ 异步删缓存’保证 MySQL 和 Redis 一致性，核心流程是：

1. 开启 MySQL 本地事务，先更新账户余额，再写入本地消息表（记录需删除的缓存 Key），保证两个操作原子性；
2. 定时任务每 5 秒扫描本地消息表，将‘待发送’的消息发送到 MQ；
3. MQ 消费端监听消息，执行 Redis 缓存删除操作，失败则重试 3 次（指数退避）；
4. 所有失败场景（MQ 发送失败、删缓存失败）都会标记状态并触发告警，缓存过期时间作为最后兜底，最终保证分钟级的强一致性。”

使用原因：金融场景是允许交易不要强一致性的，只需要最终一致性；

### 1. 通用回答框架

“解决 MySQL 和 Redis 一致性问题，核心是权衡‘一致性’和‘性能’，我会根据业务场景选择方案：

- 普通读多写少场景（如商品详情）：用 Cache Aside 方案，先更 MySQL 再删缓存，配合延迟双删 + MQ 重试 + 缓存过期时间，保证秒级最终一致；
- 核心金融场景（如支付金额）：用分布式事务 + MQ 异步删缓存，通过本地消息表保证消息必发，重试 3 次失败则告警，保证分钟级最终一致；
- 所有方案都以‘数据不脏’为核心，缓存过期时间作为最后兜底，既保证性能，又控制实现复杂度。”

### 2. 项目落地案例

“在医疗手术麻醉系统中，我用 Cache Aside 方案存储医院科室信息：

1. 读操作优先查 Redis，未命中查 MySQL 并回写缓存；
2. 写操作先更新 MySQL，再删除 Redis 缓存，同时用 RocketMQ 重试删缓存失败的情况；
3. 缓存设置 1 小时过期，延迟双删 500ms，最终实现 QPS 提升 10 倍，一致性保障到秒级，完全满足医院的业务需求。”

## 五、总结（核心要点回顾）

1. **核心原则**：一致性≠绝对一致，99% 业务只需 “最终一致”，优先保证 “数据不脏”；

2. 主流方案

   ：

   - 普通场景：Cache Aside（删缓存 + 延迟双删 + MQ 重试 + 过期时间）；
   - 强一致场景：分布式事务 + MQ 异步删缓存；
   - 高性能场景：Write Back（非核心业务）；

   

3. **兜底方案**：缓存过期时间是所有方案的最后防线，必须设置。

这套方案覆盖了从普通业务到核心金融场景的一致性保障，既懂原理又能落地，面试时既能讲清逻辑，又能结合项目经验，让面试官认可你的实战能力。
