

# 一、DDD 核心基础概念

#### 领域驱动设计产生的原因：

#### 		单体服务使用mvc架构不能的迭代业务，所有的技术和业务都放在了service实现类或者controller中，甚至放在sql中，长此以往单体服务越来越臃肿，带来的问题就是研发的接手项目理解业务难度越来越高，修改代码也是顾前顾尾！后来就慢慢演变为拆分业务转为微服务架构，一定程度的解决了这些难点，但是时间一长，问题还是渐渐暴露！一个完整的项目一定有核心业务和非核心业务，那么拆成微服务后也会如此，这导致核心微服务随着时间也会更替的频繁到最后又成了一个臃肿了单体服务了，所以微服务并不能根本上解决这个困点，因此领域驱动设计这种结构应运而生！

#### 		领域驱动设计理念为：技术理解业务、业务驱动技术实现，他将业务拆分为各个领域，领域只做业务相关的事情，绝不依赖技术实现，比如我们的实体，只有定义get和set方法，并不能看出业务所在，我们称这种实体为残血版实体，真正的实体应该是不能直接修改值，而是通过自己定义的业务方法来修改值，这样的实体对象才能称之为高内聚低耦合的表现形式！

## 1. 分层架构（标准 4 层 + 依赖铁律）

### 四层定义 & 职责

![DDD](%E9%A2%86%E5%9F%9F%E9%A9%B1%E5%8A%A8%E8%AE%BE%E8%AE%A1/DDD.png)

#### 基本原则：上层只能调用下层，基础层可以被所有层调用，也可以主动调用应用层以及领域层的仓储接口（仓储接口直接对应的基础层的数据仓储实现类，不算设计业务，只是操作数据库表数据）

|      层级      |       英文       |                          核心职责                          |                          能否写业务                          |
| :------------: | :--------------: | :--------------------------------------------------------: | :----------------------------------------------------------: |
| 接口 / 接入层  |   `interfaces`   | 接收 HTTP/MQ/RPC、参数解析、组装统一返回`Result`、路由转发 |  ❌ 纯接入，无业务,只能做空参数校验、以及网络方面的封装调用   |
|     应用层     |  `application`   |         流程编排、事务控制、调用领域、拼接业务流程         | ⚠️ 只编排，不写核心规则，Repository查询聚合、Repository保存聚合 = 属于【数据存取IO】，归应用层管 |
| 领域层（核心） |     `domain`     |         封装业务规则、状态流转、核心校验、领域行为         |  业务逻辑、状态变更 = 属于【领域行为】，归 DomainService 管  |
|   基础设施层   | `infrastructure` |       数据库、MQ、缓存、第三方接口、MyBatis、实现类        |                         ❌ 纯技术实现                         |

### 依赖强制规则（红线，不能破）

1. 从上往下依赖：`interfaces → application → domain`

2. 基础设施只能

   被调用：

   - `domain.repository(接口)` → 由`infrastructure.persistent`实现
   - MQ 监听器 / DAO 只调`application`，不直接碰领域核心

   

3. **下层绝对不能依赖上层**（domain 绝不认识 Result、Controller、PO）

4. ##### 接口层职责为接受dto参数、调用应用服务、将应用服务返回的聚合根转为vo；应用层职责为服务编排+dto转为聚合根，基础层的persistent职责为调用持久层服务，将聚合根和pojo相互转换！

5. ####  **Domain 层必须绝对纯洁，不能依赖数据库、仓储、IO**

   Domain（领域层）是**业务的灵魂**，它只关心：

   - 规则
   - 逻辑
   - 状态变更
   - 业务流程

   它**不应该知道数据是从 MySQL 来的，还是从缓存来的**。

6. 持久层sql返回的对象不一定是查询单表PO，正常来说返回聚合pojo也可以，po只是pojo的一种特殊情况，透彻点讲指DDD下不是只允许单表查询，而是区分业务将几个强关联同事务表聚合为一个pojo返回。

   什么是聚合对象？

      	比如订单表和商品表，我们经常查询订单详情，一定会联查join这两个表的，但是这俩表属于强关联吗？不是的，不能靠有没有外键来判断是不是同一聚合，要看事务看生命周期，比如用户基础信息表和用户余额表、又比如订单表和订单明细表，他们不止是通过外键关联，更是业务上紧密结合，没有用户基础信息哪来的余额信息，没有订单信息哪来的订单明细，换个角度思考其实这两个表完全可以合在一起，只是因为对应关系是一对多或者需要动静分离才被迫将他们分开设计的，它们的关系是同生命周期的，而订单没有了，商品表仍然可以存在，并没有强关联。

## 2. 领域核心对象（必背 + 存放位置）

### ① 聚合根 Aggregate Root

- 位置：`domain.aggregate` 
- 特征：**唯一 ID、生命周期、业务行为、管理子实体**
- 权限：仓储只操作聚合根，是整个聚合的唯一对外入口
- 业务：所有跨子实体的校验、状态流转、核心限制全放这

### ② 子实体 Entity

- 位置：`domain.entity`
- 特征：有 ID，但**不能独立存在**，归聚合根管理
- 限制：无独立仓储，只能通过聚合根增删改查

### ③ 值对象 VO（Domain VO，非前端 DTO）

- 位置：`domain.vo`
- 特征：**无 ID、不可变、靠内容相等**（金额、地址、单号）
- 注解：`@Getter + @AllArgsConstructor + @EqualsAndHashCode`
- 限制：只能依附聚合根 / 实体，不单独存库

### ④ 枚举 Enum

- 位置：`domain.enums`
- 特征：订单状态、支付状态、业务码（纯业务枚举）

### ⑤ 仓储 Repository

- 接口：`domain.repository`（只定义接口，入参 / 返回值 = 聚合根）
- 实现：`infrastructure.persistent`（做聚合根↔PO 转换）

### ⑥ 领域服务 DomainService

- 位置：`domain.service`
- 场景：跨多个聚合根的业务规则（单个聚合搞不定的）

### ⑦ 应用服务 ApplicationService

- 位置：`application.service`
- 场景：编排多个领域能力、加事务、组装流程

### ⑧ 数据映射对象

- PO（数据库持久化对象）：`infrastructure.po`，带 MyBatis 注解，纯数据
- DTO / 前端 VO：`interfaces.dto`，接口入参出参，组装 Result 返回

# 二、完整标准包结构图（可直接落地建包）

```
com.xxx.project
├── interfaces                # 接入层
│   ├── controller            # HTTP接口
│   ├── mq/consumer          # 临时适配（极简项目）
│   ├── dto                  # 前后端交互入参/出参
│   └── common               # 统一Result、全局异常、枚举（接口用）
├── application               # 应用层
│   ├── service              # 应用服务（流程编排）
│   └── dto                  # 应用内部流转DTO
├── domain                    # 领域核心（最干净）
│   ├── aggregate            # 聚合根（Order、User）
│   ├── entity               # 子实体（OrderItem）
│   ├── vo                   # 领域值对象（Money、Address）
│   ├── enums                # 业务状态枚举
│   ├── repository           # 仓储接口（仅抽象）
│   └── service              # 跨聚合领域服务
└── infrastructure            # 基础设施层
    ├── persistent           # 仓储实现类（聚合根↔PO转换）
    ├── mapper               # MyBatis-Plus Mapper接口
    ├── po                   # 数据库持久化对象
    ├── mq                   # MQ配置+监听器
    ├── cache                # 缓存实现
    └── thirdparty           # 第三方接口调用
```

# 三、全链路数据流转图

1. 前端 HTTP → Controller (interfaces) → 解析 DTO → 调用 ApplicationService
2. ApplicationService → 调用聚合根 / 领域服务 → 执行业务规则
3. 聚合根需要存库 → 调用 Repository 接口 (domain)
4. Repository 实现类 (infrastructure) → 聚合根转 PO → Mapper 操作 DB
5. 查库反向：PO → 聚合根 → 应用组装 DTO → Controller 包装 Result 返回
6. MQ 消息：监听器 (infrastructure) → 直接调用 ApplicationService，不碰领域

# 四、硬核强制限制（踩坑红线，必须遵守）

## 1. 层级限制

❌ domain 不能依赖：PO、MyBatis、MQ、Result、Controller、DTO

❌ application 不能写核心业务校验，只能编排流程

❌ interfaces 不能写任何业务逻辑，只做参数转发、结果封装

❌ 基础设施不能直接调用 domain 实体 / 领域服务，只能调 application

## 2. 对象使用限制

1. Repository 接口：**只能入参 / 返回聚合根，禁止子实体、PO、DTO**
2. 领域实体 / 聚合根：只用`@Getter`，禁用`@Data/@Setter`；改字段只能靠**构造方法 + 业务方法**
3. 聚合根↔PO 转换：**只允许在 infrastructure.persistent 实现类中做**，其他层严禁转换
4. 统一返回 Result：只存在 interfaces 层，domain/application 严禁出现
5. 业务校验：基础字段校验放构造方法，跨状态 / 跨明细校验放**聚合根**，跨聚合放领域服务

## 3. 技术组件限制

- MyBatis-Plus、XML、Mapper、PO：全在 infrastructure，领域零感知
- MQ 监听器：只做消息解析，不写业务，直接调用应用服务
- 所有第三方 SDK、缓存、工具类：全收进基础设施

## 4. 最易犯错禁令

1. 禁止把业务枚举、领域规则放到 interfaces/infrastructure
2. 禁止直接 new PO 传给领域，禁止领域返回 PO 给上层
3. 禁止子实体单独建 Repository，所有子实体 CRUD 走聚合根
4. 禁止在 Controller/Application 里写 if 判断业务状态（业务规则下沉领域）

------

# 五、极简记忆口诀

1. 四层从上到下：接入→编排→核心→技术
2. 业务规则锁死领域，技术实现全放基础
3. 仓储接口归领域，实现转换归基础
4. 聚合根管一切，子实体不独立，值对象不可变
5. Result 只给前端看，领域纯净不沾接口
