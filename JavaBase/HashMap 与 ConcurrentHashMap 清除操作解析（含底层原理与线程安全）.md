# HashMap 与 ConcurrentHashMap 清除操作解析（含底层原理与线程安全）

在讲解清除操作的区别前，先明确核心结论：**HashMap 是线程不安全的非并发容器，ConcurrentHashMap 是 JDK 专为高并发场景设计的线程安全容器，二者清除操作（清空/删除元素）的底层逻辑、线程安全特性完全不同**。

## 一、先理清核心概念：底层结构与线程安全基础

### 1. HashMap 底层原理（JDK 1.8）

HashMap 核心结构是「数组 + 链表 + 红黑树」：

- 数组（Node[] table）：哈希桶，元素按哈希值定位到对应桶位；

- 链表/红黑树：解决哈希冲突，桶内元素超过 8 个转为红黑树，低于 6 个转回链表；

- 无任何线程安全保障：所有操作（增删改查、清空）均未加锁，多线程操作时会出现数据错乱、死循环（JDK 1.7 头插法，在jdk1.8后采用尾插法可以避免并发时候出现死循环的情况）、元素丢失等问题。

### 2. ConcurrentHashMap 底层原理（JDK 1.8）

ConcurrentHashMap 抛弃了 JDK 1.7 的「分段锁（Segment）」，改用「数组 + 链表 + 红黑树 + CAS + 同步锁（synchronized）」：

- 核心结构和 HashMap 一致（数组+链表+红黑树），但通过「细粒度锁」保障线程安全；

- 锁粒度：锁定**单个哈希桶（桶级锁）**，而非整个容器，并发性能远高于 HashTable（全表锁）；

- 辅助手段：CAS 操作（无锁）处理桶初始化、节点插入等轻量操作，进一步提升并发效率。

## 二、清除操作的具体区别（清空/删除元素）

清除操作分两种场景：**清空整个容器（clear）**、**删除单个元素（remove）**，二者差异如下：

### 1. 清空操作（clear()）

|特性|HashMap|ConcurrentHashMap（JDK 1.8）|
|---|---|---|
|底层逻辑|遍历数组，将所有桶位置为 null|遍历数组，逐个桶位加锁（synchronized），将桶内元素置空|
|线程安全|不安全：多线程清空时可能部分桶未置空，或清空后其他线程插入元素导致数据不一致|安全：但「弱一致性」——清空过程中其他线程仍可读写，最终所有桶会被置空，但遍历过程中可能看到部分未清空的元素|
|性能|快（无锁），但多线程下不可用|并发安全，性能优于 HashTable（细粒度锁）|
### 2. 删除单个元素（remove(key)）

|特性|HashMap|ConcurrentHashMap（JDK 1.8）|
|---|---|---|
|底层逻辑|定位桶位 → 遍历链表/红黑树 → 找到元素后删除，无锁|定位桶位 → 加锁（synchronized）锁定桶节点 → 遍历删除 → 释放锁，全程 CAS 校验|
|线程安全|不安全：多线程删除可能导致链表断裂、元素丢失，或 modCount 不一致触发 ConcurrentModificationException|安全：锁定单个桶，其他桶仍可并发操作，删除过程中不会被其他线程干扰|
|异常风险|多线程遍历删除易抛并发修改异常|弱一致性遍历，不会抛并发修改异常|
#### 核心差异说明

- HashMap 的 remove：直接操作数组/链表，无任何同步措施。比如线程 A 正在删除桶 1 的链表节点，线程 B 同时遍历桶 1，可能导致链表指针错乱，甚至死循环。

- ConcurrentHashMap 的 remove：

    1. 通过哈希值定位到目标桶位；

    2. 用 `synchronized` 锁定该桶的头节点（仅锁定当前桶，不影响其他桶）；

    3. 遍历桶内元素找到目标节点并删除；

    4. 释放锁，同时通过 CAS 更新容器大小；

    5. 整个过程中，其他线程可正常操作其他桶，仅当前桶被短暂锁定，并发效率高。

## 三、ConcurrentHashMap 为什么线程安全？

ConcurrentHashMap 的线程安全并非单一手段，而是「组合拳」，核心保障如下：

### 1. 细粒度锁：桶级锁定（JDK 1.8）

- 摒弃 JDK 1.7 的分段锁（Segment），改用「单个哈希桶」作为锁粒度：

    - 每次只锁定操作的目标桶，而非整个容器；

    - 不同桶的操作可完全并发，仅同一桶的操作串行，大幅提升并发性能。

### 2. CAS 无锁操作：轻量同步

- 对「桶初始化、节点状态更新、大小统计」等轻量操作，使用 CAS（Compare-And-Swap）替代锁：

    - CAS 是硬件级别的原子操作，通过「比较-交换」保证操作的原子性；

    - 比如初始化桶时，多个线程竞争，只有一个线程能通过 CAS 成功初始化，其他线程直接使用已初始化的桶，无锁冲突。

### 3. 不可变的节点设计

- ConcurrentHashMap 的节点（Node）默认是 `final` 修饰的：

    - key 和 hash 值不可修改，避免多线程下元素被篡改；

    - 仅 value 可修改，且修改时需加锁，保证原子性。

### 4. 弱一致性的遍历机制

- 遍历采用「快照式」迭代器，不抛出 ConcurrentModificationException：

    - 遍历过程中，其他线程的修改操作不会影响当前遍历（看到的是遍历开始时的快照）；

    - 保证遍历安全，同时不阻塞写操作。

### 5. 关键操作的原子性保障

- 所有修改操作（增/删/改）均通过「锁 + CAS」保证原子性：

    - 比如删除元素时，先加锁再删除，避免多线程同时修改同一桶；

    - 大小统计（size()）通过 CAS 累加各桶的计数，保证结果准确。
    
    ## ConcurrentHashMap 线程安全的核心关键点：
    
    1. **细粒度锁 + CAS 结合**：空桶用 CAS 无锁插入，非空桶用 `synchronized` 锁定单个桶头节点，既保证安全又减少锁竞争；
    2. **volatile 保证可见性**：数组和节点的核心字段（val、next）用 volatile 修饰，确保多线程间数据即时同步；
    3. **扩容隔离 + 多线程协助**：扩容不阻塞读写，多线程可同时协助扩容，避免单线程扩容的性能瓶颈；
    4. **无锁读操作**：读操作全程无锁，依赖 volatile 保证可见性，兼顾性能和一致性。

## 四、核心区别总结

|维度|HashMap|ConcurrentHashMap|
|---|---|---|
|线程安全|不安全（无任何同步措施）|安全（桶级锁 + CAS + 不可变节点）|
|清除操作锁粒度|无锁|单个哈希桶（细粒度锁）|
|并发修改异常|遍历删除易触发|弱一致性遍历，不触发|
|性能（高并发）|高但数据不安全|并发安全且性能高（仅锁定单个桶）|
|适用场景|单线程/无并发修改的场景|多线程高并发读写的场景|
